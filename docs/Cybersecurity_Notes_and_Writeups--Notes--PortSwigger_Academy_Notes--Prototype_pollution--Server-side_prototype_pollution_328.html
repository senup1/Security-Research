Server-side prototype pollution


Server-side prototype pollution is much harder to detect than client side.

POST or PUT requests that submit JSON data are good targets as they often respond with a JSON representation fo the new or updated object.

Any features that involve updating user data are worth investigating as these often involve merging the incoming data into existing user objects.

Another approach to detect prototype pollution vulnerability is to attempt injecting potential server configuration options and look for behavior changes.

In some cases, when there are errors, JavaScript may issue a generic HTTP response containing an error object in JSON format.

Another potential access point is JSON spaces. The Express framework provides a json spaces option to customzie the number of spaces used to indent any JSON data in the response. This option is often left undefined to use the default value making it susceptible to prototype pollution. This has been patched in versions 4.17.4 and newer.

Charset override is another potential entry point. Express servers may implement modules to parse the body of incoming requests.

If you find an object with properties that are visible in the response, you can probe for sources. For example: Add an arbitrary UTF-7 encoded string to a property that's reflected in a response. Since servers don't typically use UTF-7 encoding, the string should appear in the response still encoded. Then, try to pollute the prototype to use UTF-7 “__proto__": {"content-type":"application/json; charset=utf-7"}. Next, repeat the first request. If the result contains the decoded version of the UTF-7 encoded text, the prorotype pollution was successful.
