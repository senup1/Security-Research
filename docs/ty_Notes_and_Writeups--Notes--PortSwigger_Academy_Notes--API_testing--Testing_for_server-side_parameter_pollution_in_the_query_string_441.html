Testing for server-side parameter pollution in the query string


To test for server-side parameter pollution, place query syntax characters like #, &, and = and observe the application responses.

Here is an example of what can be vulnerable to server-side parameter pollution:

GET /userSearch?name=peter&back=/home is the request on the client side

GET /users/search?name=peter&publicProfile=true is what occurs server side before being sent back to the client

The goal is to try to malform the rest of the server-side request using the syntax characters. As usual make sure to try URL encoded and regular characters.

One method is to send a # to truncate the rest of the server-side request.

Another is to append the & with some arbitrary extra parameter to see if the extra parameter is parsed or not.

You can also use discovered hidden parameters along with & as another way of altering the request.

You can even try overriding the existing parameter by using & and sending the same parameter with a different value. Results for this will vary depending on the framework being used.

A RESTful API may place parameters in the URL path instead of as a query string.

For example /api/user/jimmy rather than /api/user?user=jimmy.

It is also possible they use queries client side and RESTful server side so attempt path traversal as well.
