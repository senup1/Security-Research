Constructing a web cache deception attack


Constructing a web cache attack typically involves the following steps:

  1. Identify target endpoint that returns dynamic response containing sensitive information. Prioritize GET, HEAD, or OPTIONS supported endpoints as modifying methods like POST are typically not cached as they are dynamic.
  2. Identify a discrepancy in how the cache and origin server parse the URL path.
  3. Craft a URL that abuses the discrepancy to trick the cache into storing a dynamic response.

Remember to change requests by modifying any query parameter on each send as if this is not done, cached responses might be served rather than origin server content. This can be automated using the Param miner extension in burpsuite

The X-Cache header provides details on the cache status of the resource being accessed.

Cache-Control may also be relevant but doesn't always indicate caching is occuring.

Response time differences in requests can also be a way to detect accessing a cached resource vs directly from the origin server.
