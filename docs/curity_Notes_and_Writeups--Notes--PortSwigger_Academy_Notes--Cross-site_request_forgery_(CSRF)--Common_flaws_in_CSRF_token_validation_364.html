Common flaws in CSRF token validation


Some applications may properly validate CSRF tokens from POST requests but not in GET requests. In this case switching POST to GET in the request may allow an attacker to bypass the check.

Some applications validate the token correctly if it is present but skip the validation if the token is omitted.

Some applications do not validate that the token belongs to the same session as the user who is making the requst. Meaning anyone can use someone elses session key to bypass the verification as there is just a list of valid session keys kept on the server. It may be possible that each key only allows one use so dropping the request when gathering the key may be necessary.

Some applications will implement sessions and CSRF tokens in a way where they are not integerated together. This may result in errors in manual connection with the cookie being checked for a matching CSRF key but not checked for a matching session key.
