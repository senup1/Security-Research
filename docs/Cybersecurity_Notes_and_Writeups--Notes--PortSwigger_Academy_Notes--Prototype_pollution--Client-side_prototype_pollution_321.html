Client-side prototype pollution


Finding prototype pollution is a case of trial and error.

  1. Try injecting an arbitrary property via the query string, URL fragment, and any JSON input. Ex: website.com/?__proto__[foo]=bar
  2. In the browser console, inspect Object.prototype to see if pollution was successful. Ex: Object.prototype.foo //prints bar if successful and undefined otherwise
  3. If the property was not added, try dot notation instead
  4. Repeat with each potential source

Once a vulnerable source is identified, the next step is to find a usable gadget to use for an exploit.

  1. Look through the source code and identify any properties that are used by the application or any libraries that it imports.

Some may attempt to block the access of prototype pollution by stripping any properties with the __proto__ key from user controlled objects. This approach is flawed as there are other ways of referencing Object.prototype without relying on __proto__.

Unless its property is set to null, every object has a constructor property which references the constructor method that was used to create it.

Every constructor function has a prototype property.

Constructors can be referenced like so: myObject.constructor. So its prototype would be myObject.constructor.prototype.

Another method one might try is santizing input. If done correctly this may be a good solution. However with weak sanitization there is still room for exploitation. An example would be a lack of recursive checking if the only santization was removing __proto__ from input. Ex: __pro__proto__to__ would still go through as __proto__ since only one of the __proto__s would be stripped.
