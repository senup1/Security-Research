Using normalization discrepancies


If the cache and origin server don't have the same/proper decoding rules, it could introduce vulnerabilities.

For example, if there are rules to only allow contents in specified folders like /static to be cached but the URL decoding rules don't handle dots or slashes properly, something like /static/..%2f<sibling directory> could be used to cache a page containing sensitive information.

A method to check for this vulnerability is to append /<arbitrary string>/..%2f<current directory from original URL> to the base URL.

If the response matches the base response, the %2f was decoded properly and encoding the dot can be attempted next.

If the response doesn't match the base response, this indicates that the path was interpereted with the %2f.

The initial goal should be to find a directory, file, or page, that is known to be cached to test with.

Note that for this method static directories are the target instead of specific file extension types. Although the file extension type vulnerability may assist.

In other words, to successfully utilize normalization discrepancies, find a directory used for static files that has caching enabled on the folder. So the response should contian X-cache header info.

Next, insert <arbitrary string>/../ between the static resource holding folder and the resource this specific URL is accessing.

If it is still funcitoning properly, the next step is to remove the ../ portion of the earlier insertion.

The response should now result in 404 not found but still contain the X-Cache header that updates if it is vulnerable.

Next, remove everything after the static folder and use directory traversal to navigate to a resource containing sensitive information.

Now replace the slash with its url encoded alternative %2F and see if it is accessible and visible in the URL which would imply there is normalization descrepancy between the origin server and the cache.

Then, use earlier found delimeter with arbitrary string to cache bust and cache the resource.
